# 项目规则

## 工作流程
- **新对话启动必做**：在开启新对话时，首要任务是依次读取并分析：
  1. 现有核心代码库
  2. `e:\机器人比赛资料\第三代机器人\机器人32控制\FreeRtos\底盘说明pdf` 文件夹下的 PDF 协议与说明文档
  3. 项目规则文件 `CLAUDE.md`
  4. 上下文传承文件 `AI_CONTEXT.md`
- **修改代码前必做**：必须先说明详细修改计划，解释要做什么、怎么做、改哪些文件，等待用户回复“同意”或下达指令后方可执行修改。

## 编码规范
- 所有 C/H 文件统一使用 **UTF-8（无BOM）** 编码
- Keil MDK 需设置：Edit → Configuration → Encoding → UTF-8
- 与正点原子代码风格保持一致
- 写入文件时使用 UTF-8 编码

## PDF 读取工具配置
- **工具**: Poppler (pdftoppm)
- **安装位置**: C:\poppler\poppler-24.08.0\Library\bin
- **环境变量**: 已添加到 Windows User PATH
- **使用方法**: 直接使用 Read 工具读取 PDF 文件，指定 pages 参数
- **注意**: 如果 pdftoppm 不可用，按以下步骤安装：
  1. 下载 Poppler: `curl -L -o /tmp/poppler.zip https://github.com/oschwartz10612/poppler-windows/releases/download/v24.08.0-0/Release-24.08.0-0.zip`
  2. 解压到 C:\poppler: `mkdir -p /c/poppler && cd /c/poppler && unzip -q /tmp/poppler.zip`
  3. 添加到 PATH: `powershell.exe -Command "[Environment]::SetEnvironmentVariable('Path', [Environment]::GetEnvironmentVariable('Path', 'User') + ';C:\poppler\poppler-24.08.0\Library\bin', 'User')"`
  4. 重启 VSCode 使环境变量生效

## 项目信息
- 平台：STM32F407ZGT6
- RTOS：FreeRTOS
- 通信：CAN (500Kbps, 中断+队列模式)
- IDE：Keil MDK-ARM

## 开发规则（工程化指导）

### 1. 开发流程规则
- **需求确认优先**：每个模块开发前，先讨论需求和接口设计，你确认后再动手
- **小步迭代**：每次只实现一个小功能，测试通过后再进行下一步
- **代码审查**：我写完代码后，先向你解释设计思路和关键代码，你理解后再写入文件
- **测试驱动**：每个模块写完后，先写测试代码验证功能，再集成到主程序

### 2. 代码规范
- **遵循正点原子风格**：变量命名、注释格式与现有代码保持一致
- **UTF-8编码**：所有C/H文件使用UTF-8（无BOM）编码，中文注释
- **模块化设计**：每个功能独立成模块（.c/.h），接口清晰
- **防御性编程**：参数检查、边界检查、错误处理必须完善
- **注释完整**：函数头注释、关键逻辑注释、复杂算法说明
- **修改后自查**：每次修改代码后，必须立即重新读取文件检查是否有乱码或错误
- **零容忍原则**：任何问题（包括编码错误、协议错误、逻辑错误等）都必须立即修复，除非用户明确说不需要修复
- **协议实现强制规则**：
  - 实现任何通信协议前，必须先完整阅读协议文档（PDF）
  - **重要：如果有任何不确定的地方，必须及时查阅 `e:\机器人比赛资料\第三代机器人\机器人32控制\FreeRtos\底盘说明pdf` 文件夹下的文档**
  - 协议文档中的每个字段（单位、字节序、数据类型、取值范围）必须严格对照实现
  - 代码注释中的单位必须与协议文档完全一致
  - 编码/解码函数实现后，必须用协议文档中的示例数据验证
  - 禁止凭记忆或猜测实现协议，必须逐字对照文档

### 3. 架构规则
- **分层解耦**：驱动层、抽象层、应用层严格分离
- **接口抽象**：底盘驱动提供统一接口，方便后续更换底盘型号
- **数据结构优先**：先设计好数据结构，再实现功能
- **状态管理**：使用状态机管理系统状态，避免混乱的if-else

### 4. 安全规则
- **超时保护**：所有通信必须有超时检测
- **故障处理**：底盘故障、通信异常必须有应对措施
- **紧急停止**：任何异常情况下能立即停止底盘
- **参数校验**：速度、角度等参数必须限幅检查

### 5. 调试规则
- **日志分级**：ERROR/WARNING/INFO/DEBUG，方便定位问题
- **状态可视化**：LCD显示关键状态，LED指示运行状态
- **调试开关**：通过宏定义控制调试信息输出
- **版本管理**：每个模块标注版本号和修改记录

### 6. 文档规则
- **接口文档**：每个模块的头文件就是接口文档，注释要详细
- **协议文档**：上位机通信协议单独文档说明
- **测试记录**：关键测试结果要记录（速度精度、响应时间等）

### 7. 教学规则（重要！）
- **讲解优先**：我先讲解设计思路、为什么这样做、有哪些坑
- **知识点说明**：涉及的技术点（如CAN协议、运动学、RTOS）我会解释清楚
- **举一反三**：不只是完成功能，还要让你理解原理，能自己扩展
- **问题引导**：遇到选择时，我会给出多个方案，分析利弊，让你决策

### 8. 协作规则
- **你是主导**：技术方案由你最终决定，我提供建议和实现
- **及时反馈**：如果我的方案不合适，随时提出，我们调整
- **实际优先**：理论让位于实际，能跑起来、能用好才是目标
- **保持沟通**：不懂就问，我会解释到你理解为止

### 9. 信息传递原则
- **分段提问**：一次只问1-2个关键问题，不要一次性抛出大量问题
- **信息密度适中**：每次回复的信息量要让用户能够消化和处理
- **可操作性**：给出的信息必须是用户当前能够立即执行的
- **避免信息过载**：不要一次性给出过多方案、参数、选项
- **回复结构**：当前问题（1个）→ 简要说明（2-3句）→ 具体方案（≤3个选项）→ 下一步行动
- **进度反馈**：执行长时间操作时实时告知进度，避免让用户以为卡住了
