- **控制对象**: RMP-WL100 四驱四转向轮式底盘

## 2. 最近一次的关键变更 (2026-02-28 协议大修与通信锁定)
在本次开发中，我们对比了曼大智能底盘的官方厂规 PDF 和其 ROS2 驱动源码，并基于此对 FreeRTOS 端的 `chassis_driver.c/h` 进行了**重构性修正**，彻底解决了协议字段错位和安全隐患。

### 2.1 修复的大坑 (重点提防区)
1. **0x402 Vz(角速度) 乘数错误 (高危隐患已修复)**：
   - 原代码中，下发指令的 `Vz` 被错误地乘以了 `4000.0f`。
   - **真相**：手册中明确规定所有速度(Vx, Vy, Vz)和角度的编码乘数一律为 `1000.0f`（即将浮点数放大一千倍转换为 int16）。现已将 `Vz * 4000` 修复为 `Vz * 1000`。
   - **防飞车保护**：同时在 `encode_motion_ctrl` 中加入了**软防飞车限幅机制**（强制裁切绝对值超过 `2.0` 的速度值，防止底层数组溢出）。

2. **0x100 系统状态 字节不对齐的修复**：
   - 修改前的代码错把 `Byte[0]~[1]` 当成了电压，把 `Byte[6]` 当成了 SOC。
   - **更新后正确的映射：**
     - `Byte[0]`: 系统状态
     - `Byte[1]`: 控制模式
     - `Byte[2]`: 外设状态
     - `Byte[3]`: **SOC (电量)**
     - `Byte[4]~[5]`: **电压** (单位10mV，注意是大端)
     - `Byte[6]~[7]`: **电流** (单位10mA)

3. **大小端机制的最终确立 (重要共识)**：
   - 官方手册明确了所有 16 位整型数据均遵循 **大端序 (高字节在前)**。
   - 我们抛弃了 ROS2 源码里因为 x86 平台妥协而写的丑陋“小端数组倒置硬拷贝”大法，坚持在 STM32 中使用标准移位解码：`val = (data[0] << 8) | data[1];`。这确保了代码极高的跨平台可读性和架构纯洁性。

4. **补上了丢失的 0x103 故障检测协议**：
   - 完整实现了底盘四个驱动器故障/失联报告的解析与结构体托管。

## 3. 架构梳理
- **驱动层 (`Drivers/BSP/CAN/can.c/.h`)**: 负责底层 CAN 收发。中断中通过 `xQueueSendFromISR` 将报文丢到 OS 队列中。
- **抽象层 (`Drivers/BSP/CHASSIS/`)**: 
  - `chassis_protocol.h` (纯数据协议)、`chassis_driver` (负责报文的大端编解码及状态暂存)。
- **应用层 (`User/chassis_test.c`)**:
  - `chassis_ctrl_task`：控制逻辑（下发 0x402 等测试）。
  - `chassis_rx_task`：阻塞监听 CAN 队列，解码报文并更新状态。
  - **新引入的 Mutex 锁机制**：为了防止因为多任务并发调用 `printf` 和中断导致串口输出断行/互相污染，项目中已加入 `safe_printf` 宏包装和 `g_printf_mutex` 的信号量互斥锁。
